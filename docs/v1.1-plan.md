# Orchestration Plugin — v1.1 Plan

**Branch:** `feature/workspace-discovery-and-resume`
**Theme:** Improve the onboarding experience and establish a scalable, **deterministic** basis for workspace configuration & settings management — *before* adding more agents.

**Guiding principle:** maximize determinism at every step. Mechanical work is code (`lib/*.mjs` + schema validators), not prose an LLM interprets. **JSON is the source of truth; markdown is a rendered view that is never parsed.**

---

## Decisions (locked)

### 1. Version
- `plugin.json` → **1.1.0** (was mistakenly committed as 1.3.0). ✅ done.

### 2. Onboarding readiness check
- A plugin **`SessionStart`** hook (matcher `startup`) runs `node ${CLAUDE_PLUGIN_ROOT}/lib/onboarding.mjs`.
- The script reads the hook JSON from stdin (`.cwd`), reuses `detectTopology()`, and checks **deterministically** whether the folder is configured: the profile file exists **and** passes `validateWorkspace()`. No markdown is read.
- If not ready **and** the readiness check is enabled → emit `hookSpecificOutput.additionalContext` prompting the model to *offer* `/orchestrate-config init` (resume wording if a `*.draft` exists). Otherwise stay silent.
- **Hard guarantee:** the hook can never throw, never block, never print an error — worst case it exits 0 silently.

### 3. Settings — two layers, session-scoped
| Setting | Meaning | Default (post-install) | Mechanism |
|---|---|---|---|
| 1. Plugin on/off everywhere | install-level enablement | ON | **Claude Code's built-in `enabledPlugins`** — not reinvented |
| 2. Readiness check on/off | run the "is this folder configured?" check | ON | plugin-owned config, cascade below |

- Cascade: global `~/.claude/orchestration/config.json` ← overridden per-workspace by `<root>/.claude/orchestration/config.local.json`.
- **Session-scoped:** settings are read once at `SessionStart` and fixed for the session. A mid-session change is *not* applied live — the plugin tells the user to start a new session. This is documented behavior.

### 4. Profile knowledge links (per member)
- New `knowledge` object per member: fixed slots (`claude_md`, `skills`, `rubrics`, …), each a resolved path **or `null`** (null = missing link OR missing file — consumer treats both the same).
- Plus `extra`: a free-form JSON object for user-defined custom links (readable by agents, or pointed at by the user).

### 5. Provenance split (makes `update` correct by construction)
- **Detected facts** — always re-derivable from disk; never the durable truth.
- **Overrides** (`overrides.local.json`) — the *only* durable human-authored artifact; what `set` writes.
- **Profile** (`workspace.json` / `.local.json`) — a **derived** artifact = `detected ⊕ overrides`, regenerated by `init`/`update`; consumed by runs. (Mirrors how the `.md` is derived from the `.json`.)
- **`update`** = re-detect → re-apply persisted overrides → regenerate:
  - new members/gates/knowledge flow in automatically;
  - user `set` overrides are preserved by construction;
  - changed-but-not-overridden detected values flow through, reported in a "what changed" summary;
  - the only true **conflict** is a *stale override* (points at something now gone/changed) → surfaced and asked.

### 6. Agent rework
- `chuck-workspace-analyst` → **`orchestration-settings-manager`** (full absorption): one agent owns the whole config lifecycle (detection + settings management).
- Naming convention: **`chuck-*`** = domain workers (code); **`orchestration-*`** = meta/management agents (the plugin itself).

### 7. Command surface
- Umbrella **`/orchestrate-config <action>`**: `show`, `set`, `init`, `update`. No `reset` (config is gitignored/personal — nothing canonical to reset to).
- `/orchestrate` (run a ticket) stays.

### Topology
- Keep the trichotomy single-repo / monorepo / multi-repo. mono vs multi determines whether "the diff" is one baseline or one-per-member.

## Implementation phases

### Phase A — Deterministic substrate + onboarding (headline feature)
1. `lib/schema.mjs` — bump workspace schema to `@2`; validate `knowledge` (slots + `extra`); add config schema `orchestration/config@1`.
2. `lib/detect.mjs` — populate per-member `knowledge` links from disk.
3. `lib/config.mjs` — read/write/cascade Setting 2 across global + workspace; defaults ON.
4. `lib/onboarding.mjs` — deterministic readiness check + `SessionStart` output; defensive (never throws).
5. `hooks/hooks.json` — register the `SessionStart` hook.
6. `lib/profile.mjs render` — surface `knowledge` in the human view.

### Phase B — Provenance engine + guided config commands

**B-1 — the deterministic overrides engine (built & verified first, in isolation):**
7. `schema.mjs` — add `orchestration/overrides@1` + `validateOverrides`. An override file is a per-member map of `{ role?, role_reason?, gates?, knowledge? (slots + extra), notes? }`. Fixed slots/roles/gates are allowlisted; only `knowledge.extra.*` is free-form (values must be string links).
8. `lib/overrides.mjs`:
   - `apply <detected.json> <overrides.json> [--out]` — derive the profile = **detected ⊕ overrides**; emits stale-override conflicts (overrides pointing at a member no longer detected) to stderr.
   - `set <overrides.json> <member> <field> <value>` — the **deterministic structured writer** the guided command calls with fully-resolved args. `field` ∈ { `role`, `role_reason`, `gates.<key>`, `knowledge.<slot>`, `knowledge.extra.<name>`, `note` }. Re-validates every write.
   - `unset <overrides.json> <member> <field>` — remove one override (so a bad override can be cleared).
   - `validate` / `show`.
   - Provenance: `workspace.json` becomes a **derived** artifact regenerated from detected ⊕ `overrides.local.json`; the overrides file is the only durable human artifact.

**B-2 — the guided `/orchestrate-config` command (UX on top of the engine):**
9. Umbrella `/orchestrate-config <action>`: `show` / `set` / `init` / `update`.
10. **`set` is guided, never a hand-typed dotted path.** The agent reads the live profile and walks the user through menus: pick member → pick target (role / knowledge slot / add-or-edit an `extra` link / plugin setting) → contextual value step (roles enumerated; paths existence-checked; `extra` = name + link) → confirm → call the deterministic writer (`overrides.mjs set` for profile fields, `config.mjs set` for settings). The dotted address is the engine's internal contract, never the human interface.
11. `init` = detect → ask crucial decisions → seed `overrides.local.json` → derive profile. `update` = re-detect → re-apply overrides → regenerate + report what changed and any stale-override conflicts.

### Phase C — Agent rework & docs
9. Rename/rework `chuck-workspace-analyst` → `orchestration-settings-manager` (full absorption).
10. Update `WORKSPACE.md`, `ORCHESTRATION.md`, `README.md`, `marketplace.json` / `plugin.json` descriptions.
11. Retire/alias `/orchestrate-setup` → `/orchestrate-config init`.

Each phase is verified by running the `lib/*.mjs` scripts directly (node) against this repo and temp fixtures before moving on.

## Post-release follow-ups
- **Validate the guided `set` UX across real cases** after the 1.1 release (with wider user access) — the deterministic engine is fixture-verified, but the menu flow's ergonomics need real usage to confirm. Reversible knobs: menu ordering (surface-first vs member-first), `show` default verbosity, per-conflict vs batched `update` prompts.
- Release notes: `workspace@1` profiles re-prompt for `init` (no migration shim, by design); the `1.3.0 → 1.1.0` version renumber is intentional.
